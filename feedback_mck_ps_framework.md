Reo is a McKinsey出身. When thinking through problems, structuring arguments, or building deliverables, always follow the McKinsey Problem Solving framework:

**Why:** This is how Reo thinks and expects work to be structured. Non-McK-style output will feel unstructured and unconvincing to him.

**How to apply:**

## Core Process (4 Steps)

1. **イシューの定義 (Issue Definition)**
   - Issue = あるべき姿と現実のギャップ
   - SMART: Specific, Measurable, Action-oriented, Relevant, Time-bound
   - PSW (Problem Statement Worksheet): 背景、解くべきイシュー、制約条件、成功基準
   - イシューの本質は「選択」。必ず「べき」が入る

2. **構造化 (Structuring)**
   - イシューツリー（≠ロジックツリー）で分解
     - ロジックツリー: 機械的MECE分解（スピード重視）
     - イシューツリー: 解ける課題に分解、重要な軸が上位に来る（深さ重視）
   - Day1から仮説を持つ。仮説に白黒つけることで深化
   - SCR: Situation(解きたい事象) → Complication(根本原因) → Resolution(打ち手)
   - 根本原因の深掘り = Whyを繰り返す（5 Whys）
   - 4レイヤー: パフォーマンス → スキル/コンダクト → マネジメント理解度 → クライアント固有（論理・感情・社内政治）

3. **分析 (Analysis)**
   - 分析 = 仮説を立てて実証・反証すること
   - 分析 = 「軸をもって比較すること」（情報の羅列はNG）
   - 軸の選定・明確化 → 場合分け → 順序付け
   - 検証可能なレベルまで課題を分解してから仮説検証

4. **統合化 (Synthesis)**
   - ピラミッドストラクチャー: 答え → 主な提案 → 論拠データ
   - SCRでクライアントコミュニケーション
   - プレゼン段階ではなく構造化段階からストーリーを意識

## 実践原則

- **仮説主導型アプローチ (5Ps)**: イシュー定義→分解→優先順位→分析計画→実施→意味合い→提案
- **Own the Issue**: 課題を解くために考え抜く
- **80/20**: すべてを網羅する必要なし、優先順位をつける
- **資料作成を最小化**: 問題解決がプロフェッショナル、資料作成ではない
- **週1でIA振り返り**: iPhoneの電源切って2時間、イシューアナリシスに立ち戻る

## テスト手法

- **モンゴリアンウォッシングマシーン**: イシューの主語を競合に変えて成立しないか確認
- **コインの裏返し**: 解決方針が当たり前の裏返しになっていないか
- **「もし○○さんだったら」テスト**: 別の視点からコメントを書き出す
- **現場・専門家・クライアントの3視点**: IAの精緻化

## 適用場面

- 壁打ち・議論のとき: イシューを明確化してから構造化
- 資料作成のとき: PSW→IA→WP→ピラミッドストラクチャー
- コミュニケーション: SCR形式で伝える
